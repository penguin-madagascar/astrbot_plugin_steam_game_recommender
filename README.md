# Steam 游戏推荐助手

这是一个面向 AstrBot 的 Steam 游戏推荐插件。你可以直接用自然语言描述想玩的玩法、参考游戏、语言、预算、开发商或其他具体特性。默认结果来自经过 Steam 商店数据验证的候选，并会说明为什么推荐、有哪些不确定之处；如主动开启零结果兜底，未验证的模型建议会与正式结果明确分开。

普通推荐不需要 Steam Web API Key。只有读取个人游戏库、排除已拥有游戏或使用随机库内推荐时，才需要绑定 Steam 账号并配置 Key。

## 安装与配置

1. 在 AstrBot 插件管理中安装本插件；也可以把项目目录以 `astrbot_plugin_steam_game_recommender` 为名放入 AstrBot 的插件目录。
2. 手动安装时，在插件目录执行：

   ```bash
   pip install -r requirements.txt
   ```

   请确保依赖安装到 AstrBot 实际使用的 Python 环境中。

3. 在 AstrBot Dashboard 中按需设置：

   - 模型提供商：用于理解自然语言偏好、核验模糊特性和整理推荐理由。留空时会尝试使用当前会话模型。
   - 语义特征核验批量大小：每批核验的候选数，默认为 5，可设为 1–10。通常保持默认即可；减小后单批失败影响更小，但模型调用次数会增加。
   - LLM 零结果兜底：默认关闭。只有主动选择一个专用模型提供商后才会启用，详见“错误、降级与空结果”。
   - 相同查询结果复用：默认关闭，使重复查询重新尝试 Steam 发现召回和语义核验；开启后可减少重复请求。
   - 默认地区：未在命令中指定地区时使用，初始为 `CN`。
   - 默认推荐数量上限：新安装默认为 10，范围为 1–10。
   - Steam Web API Key：普通推荐不需要；游戏库过滤和 `/randomrec` 需要。可在 [Steam 官方页面](https://steamcommunity.com/dev/apikey) 申请。
   - 随机推荐门槛：只影响 `/randomrec` 所要求的最低评测数和好评率。

如需当前价格、历史最低和最近促销，请另外安装 `astrbot_plugin_steam_price_heybox`。未安装价格插件时仍可正常推荐，只是价格会显示为暂无数据。

## 开始使用

### 命令一览

| 命令 | 中文命令 | 用途 |
| --- | --- | --- |
| `/gamerec <需求>` | `/游戏推荐 <需求>` | 根据自然语言需求推荐 Steam 游戏。 |
| `/gamerec_retry [补充要求]` | `/重新推荐`、`/换一批` | 在最近一次推荐基础上换一批，或补充、修改条件。 |
| `/accountbind <SteamID64 或好友码>` | `/账号绑定` | 绑定 Steam 账号；不带参数时查看当前绑定。 |
| `/randomrec` | `/随机推荐` | 从已绑定游戏库中随机挑选一款未玩且评价达到门槛的游戏。 |

`/gamerec` 的完整形式是：

```text
/gamerec [排除已有|仅查看已有] [区域] <自然语言需求>
```

为了让命令更容易读，建议把 `排除已有` 或 `仅查看已有` 写在需求开头。两者不能同时使用。区域可以写成 `-CN`、`-US`、`-JP` 等两字母代码，也可以写国区、美区、日区、港区、台区、韩区等常用名称。

例如：

```text
/gamerec 排除已有 -US 双人合作解谜，预算 30 美元
/游戏推荐 仅查看已有 日区 适合周末通关的剧情游戏，预算 3000 日元
/gamerec 类似黑暗之魂的游戏
/gamerec 推荐 3 款支持简体中文、预算 100 元以内的合作游戏
/gamerec 模拟类游戏，要有节日活动，最好由 ConcernedApe 开发
/gamerec 即将发售的类魂游戏
```

### 推荐数量

- 新安装默认最多返回 10 款。
- 用户原文中明确写出的数量优先，例如“推荐 3 款”会请求 3 款；支持阿拉伯数字和一至十的中文数字。
- 用户要求的数量仍不会超过 Dashboard 中保存的上限，也不会超过 10。比如上限为 5 时，“推荐 8 款”最多返回 5 款。
- 如果升级前已经设置为 5 款，升级后会保留 5，不会强行改成 10。
- `/gamerec_retry` 默认继承上一次的数量；只有补充要求明确改了数量才会变化。
- 数量只从用户原文确定性读取，模型自行猜出的数量不会生效。

目标数量不是凑数要求。符合条件的候选不足时，实际返回数量会更少。

### 重复查询与缓存

插件默认关闭相同查询结果复用。即使连续发送完全相同的需求，也会重新尝试 Steam 候选发现和模糊特性核验，以减少首次网络波动或偶发模型响应对后续查询的影响。

这不等于禁用所有缓存。游戏详情、评测、商店标签、标签词表、发售状态、价格、账号和本地游戏索引等基础商店资料仍会按各自有效期缓存；实时候选发现失败时，也可以用最多 7 天的旧发现快照降级继续。需要减少重复网络和模型请求时，可以在 Dashboard 中开启相同查询结果复用。

## 怎样提出需求

### 找“类似某款游戏”的作品

```text
/gamerec 类似黑暗之魂的游戏
/gamerec 类似逃离塔科夫，但不要 PVP
```

插件会把中文名、英文名和本地化别名当作同一个参考实体尝试解析。可靠解析后，会综合 Steam 的 More Like This、多标签交集、核心标签与其他 Steam 召回结果寻找候选，而不是只凭一个宽泛类型标签。

一次查询最多处理 3 个参考游戏，每个参考游戏最多使用 3 个标题候选（包含原文名称）。超过上限时会保留优先项并给出解析提示；无法可靠判断属于哪款游戏的别名会被忽略，不会被错误绑定到另一个参考游戏。

参考游戏本身不会作为结果返回，但续作和同系列作品仍可参与推荐。基础版、重制版和续作编号会分别判断，尽量避免把标题相近的另一部作品误当成参考游戏。

如果参考名称没有找到或存在歧义，结果总览会提示一次，并继续使用其余明确条件；Steam 网络故障则会作为查询或降级问题提示，不会伪装成“参考游戏不存在”。

### 玩法、排除项和大作倾向

你可以同时写多个条件，并明确哪些是核心要求：

```text
/gamerec 必须本地双人合作，轻松解谜，不要恐怖
/gamerec 单机 3A 大作，支持手柄
```

明确的核心玩法比普通类型和热度更重要。参考游戏的前序、区分度较高的标签也会优先影响匹配；“动作”“冒险”等宽泛标签不能靠数量抵消核心玩法缺失。

“AAA”“3A”或“大作”只表示高知名度/大作倾向，插件会更重视成熟口碑和关注度，但不会据此猜测动作、RPG、开放世界等玩法，也不代表已确认制作预算。

当查询包含核心玩法时，插件不会用完全没有核心玩法直接证据的候选凑数。部分核心特征证据不足的候选仍可能返回，但会在“不推荐理由”中说明；候选不足时宁可少返回，也不会因为某款游戏热门或匹配许多宽泛标签就把它塞进结果。

### 节日活动、结婚、昼夜循环等模糊特性

有些要求不是稳定的 Steam 标签，例如“能过节”“可以结婚”“有昼夜循环”。插件会先用相关词和标签寻找候选，再用 Steam 标签、类型和商店描述核验这些具体特性；用于扩大搜索范围的相关标签本身不算满足要求的证据。

只有写成“必须”“一定要”“只接受”等明确强制措辞的特性才是硬条件：核验正常完成时，证据未知或明确不符的候选都会被移除。没有使用强制措辞、但属于本次查询核心的具体特性会按核验结果谨慎处理：在玩法相关性层级相同的候选中，已确认满足的候选优先，明确不符的候选被移除；证据未知的候选可能在已确认结果不足时补位，结果会显示一次提醒，并在每款游戏的“不推荐理由”中注明原始需求尚缺证据。可选特性只有在核验满足时才会增加匹配度；证据未知或明确不符时不会加分，并可能出现在“不推荐理由”中。

必须或核心特性会按每 20 款一轮核验，最多核验 60 款；一旦已确认满足的结果足够本次展示，就会停止后续核验。只有可选特性时只核验首轮候选。核验批次失败不会再把整批候选误判为不满足：受影响的候选会保留在候选中，不获得该特性的加分，并显示强提示及逐项不确定说明；其他成功批次的有效结论仍会正常使用，因此单个无效响应不会清空整次推荐。

核验只依据送入模型的 Steam 商店资料，不会访问攻略、Wiki、社区帖子或实际游玩内容。因此，商店页没有写明的隐藏机制可能会被判断为未知。

### 开发商和发行商

可以指定开发商、发行商，或不区分两者：

```text
/gamerec FromSoftware 开发的单机动作游戏
/gamerec Capcom 发行的合作游戏
/gamerec 最好是 Valve 的游戏
```

公司名称会用 Steam 详情中的开发商和发行商字段复核，不会因为游戏标题中出现同名词就当作匹配。即使写了“仅限”或“只要”，公司条件仍是软偏好：匹配者优先，资料未知或不匹配者会降低推荐分，但不会作为硬过滤条件，也不会跨越核心玩法的相关性顺序。

大作倾向不会自动生成公司条件，也不会使用发行商白名单。

### 语言、预算和地区

```text
/gamerec 必须支持简体中文的殖民模拟，预算 100 元
/gamerec -US online co-op under $30
```

语言、预算和公司类似，都是在核心玩法相关性相同或接近时影响排序的软偏好。使用“必须”“只接受”等措辞会带来更强的降分，但当前版本仍可能保留不满足或资料未知的候选，并在理由中说明不确定性。

未写币种时，预算按本次所选地区的常用币种理解。价格数据不足或价格插件不可用时，插件不会假装游戏满足预算；它会保留不确定性，同时继续返回玩法合格的候选。

### 未发售游戏

普通查询默认排除全部未发售游戏。只有明确写出“未发售”“即将发售”“待发售”“upcoming”等意图时，它们才会参与推荐。

未发售游戏如果已有有效真实评测，会使用真实评测。否则，无论是否已经开放预购，评价和知名度都按 60/100 的未发售先验参与排序，不会伪造评测数、好评率或真实知名度。结果会明确显示：

> 使用 60/100 未发售质量先验，仅用于排序，不代表玩家实评或实际知名度

已经发售但没有有效评测的游戏不会获得这项先验。

## 怎样理解结果

推荐通常以转发消息显示：运行通知在前，总览随后，每款游戏各占一个条目。客户端不支持转发节点时，会以相同顺序合并成普通文本。

示例：

```text
找到 3 款 Steam 游戏，按核心匹配层级及层内推荐分排列。

1. 《示例游戏》｜推荐分：86/100

推荐理由：命中合作与解谜核心特征，Steam 评测基础较稳。

不推荐理由：简体中文支持信息未知。

价格（US）：当前价 $19.99；历史最低 $9.99；最近促销 $11.99

购买链接：https://store.steampowered.com/app/123456/
```

- `推荐分` 是本次查询中的排序分，不是“你有 86% 概率会喜欢”的成功概率。
- 排序先看核心玩法是否成立，再在同等相关性中考虑辅助特征、Steam 评测可信度、关注度以及语言、公司和预算等软偏好。
- `推荐理由` 只概括已有的正向证据；`不推荐理由` 只在存在重要负面信息或不确定证据时出现，没有风险证据时会省略。
- 理由模型只负责选择已有证据，最终显示文字由这些受信证据规则化生成，因此模型自由文本不能自行加入价格、语言、玩法、口碑或风险事实；模型不可用或输出不可靠时同样使用规则化理由。
- 已识别的 Steam 标签会尽量用中文显示。找不到可靠中文名时，会使用中性描述，不直接把内部英文格式暴露给用户。
- `偏好解析提示` 中的多条提示会逐行列出，便于区分参考解析、游戏库过滤和 Steam 召回降级等情况。

同一作品的多个标准版、重制版或特别版通常只保留一个，避免结果被同一游戏的不同版本占满。续作编号和正常副标题不会被当作可随意删除的版本词。

## 账号、游戏库与价格

### 绑定账号

```text
/accountbind 76561198000000000
/accountbind 123456789
/accountbind
```

前两种分别可填写 17 位 SteamID64 或纯数字好友码；不带参数时查看当前聊天用户的绑定。绑定本身不要求 Steam Web API Key。

普通推荐不需要 Steam Web API Key。以下功能需要 Key、已绑定账号以及公开可读的 Steam 游戏库：

- `排除已有`：排除已拥有游戏及其同作品版本。
- `仅查看已有`：只在已拥有的精确 Steam 应用中推荐。
- `/randomrec`：从游戏库中游玩时长为零、详情确认为游戏本体且评价达到 Dashboard 门槛的作品中随机选择一款。

`/randomrec` 返回游戏名和简短理由，不附带推荐分、价格或购买链接。

### 区域价格

价格始终对应本次查询的地区，不会混合多个地区，也不做跨区购买建议。安装可选价格插件后可显示当前价、历史最低和最近促销；缺少价格或促销数据时会显示“暂无数据”，但不会阻止游戏推荐。

## 错误、降级与空结果

- 如果看到“偏好解析模型暂时不可用”，该错误会作为第一条独立消息发送；插件随后使用关键词规则继续推荐，所以复杂条件可能解析不完整。
- 如果偏好模型只返回了无效格式，插件会尝试修复一次；仍失败时再使用关键词规则。
- 如果看到“语义特征核验服务暂不可用”或“语义特征核验响应格式无效”，受影响的特性不会得到额外加分。技术故障不会被当成游戏明确不满足要求；相关候选会保留并附带醒目的未确认提示，已经成功核验的批次仍然有效。
- Steam 的部分搜索来源临时失败时，插件会使用仍可用的 Steam 来源和本地数据继续，并在总览提示能力下降。关键来源系统性失败时会直接提示 Steam 查询失败，建议稍后重试。
- “参考游戏未能可靠解析”表示名称确实没有可靠命中或结果有歧义；单纯的网络错误会单独报告。
- 当 Steam 查询正常但没有合格候选时，会返回“暂时没有找到满足当前条件的游戏。”这不是 Steam 不可用，可以尝试放宽核心玩法、排除项、人数或模糊特性后再查。

LLM 零结果兜底默认关闭，关闭时没有合格候选就只返回空结果说明。若确实希望在这种情况下获得更宽松的灵感，可以在 Dashboard 中为“正常零结果兜底”明确选择一个模型提供商；它不会自动借用当前会话模型。

兜底只有推荐流程正常完成但没有合格候选时才会运行，不会在 Steam 查询失败、程序错误或已经找到正式结果时触发。模型给出的名称必须与 Steam 目录中的游戏本体标题精确对应，确认失败的自由文本不会显示；名称通过确认也只代表该游戏确实存在于 Steam，不代表它满足需求。输出会单独标明“需求匹配未验证”，使用固定说明代替模型自由理由，也不会附带购买链接、价格、评测或推荐分，更不会写入换一批所使用的正式推荐记录。兜底模型不可用、格式不合格或名称无法确认时，插件仍保留原本的空结果说明。

## 数据边界

- 正式排序结果只推荐 Steam 游戏，不会把 Switch、PlayStation、Xbox 或其他平台作品伪装成已验证结果。主动开启的零结果兜底可能给出未验证灵感，但会与正式结果明确分开。
- 正式候选必须由 Steam 详情确认是游戏本体。DLC、试玩版、原声、工具和套餐不会进入最终推荐；详情无法确认时也不会猜测。
- Steam 社区标签可能缺失、顺序变化或包含玩梗标签。插件会结合多标签、参考游戏关系、商店类型和描述来降低单一错误标签的影响，但无法保证公开数据永远准确。
- 推荐不会因为某个具体示例游戏使用单独规则；参考解析、候选召回、标签证据和排序标准对所有游戏一致。
- 语言支持、开发商、发行商、发售状态、评测和商店描述以 Steam 当前公开信息为准；资料缺失会显示为未知，不会推断为满足。
- 模糊玩法特性只核验 Steam 商店中可见的文字和结构化信息，不能替代游玩体验、攻略或开发者确认。
- “AAA/大作”只代表高知名度倾向，因为 Steam 没有可靠的制作预算字段。
- 价格和促销依赖可用的区域价格数据，不保证未来售价、库存或促销安排。
- 推荐结果用于缩小选择范围，不代表购买保证；决定前仍建议查看 Steam 商店页、系统需求和近期玩家评价。
